# Tolvi Phase 2 — Server core design spec

**Status:** Draft (pending approval)
**Date:** 2026-05-10
**Owner:** Alan
**Sub-project of:** Tolvi v1 product roadmap, Phase 2 (server core)
**Reference:** `spec/tolvi-format-v1.md` (the public format contract this server implements); `docs/ARCHITECTURE.md` (the two-arm architecture this is one half of); `docs/superpowers/specs/2026-05-09-phase-0-1-foundation-design.md` (the foundation that locked the contract)

---

## 1. Goal

Ship the self-hostable Tolvi server: a TypeScript Fastify service over Postgres + pgvector that exposes the v1 HTTP API documented in `docs/ARCHITECTURE.md`. Multi-tenant via API keys at the workspace level, no user accounts. Comes up via `docker compose up` with zero external service dependencies (Ollama for embedding bakes in as a sidecar; Anthropic API key is the only optional external requirement, gated to `/v1/ask`).

After this phase merges, a self-hoster can:
1. Run `docker compose up`
2. POST a `tolvi-format-v1` doc to `/v1/documents` with an API key
3. Search it semantically via `/v1/search`
4. Get a cited synthesized answer via `/v1/ask`

## 2. Non-goals

Explicitly out of scope:

- Go CLI implementation (Phase 3 — runs in parallel)
- Async ingest / worker process (sync inline is enough for v1; revisit when batch loads grow)
- Streaming `/v1/ask` (SSE) — batch only for v1
- Per-workspace rate limits (error code reserved; enforcement deferred)
- User accounts, OIDC, SSO (Phase 9)
- Web dashboard (Phase 9)
- Lexical / hybrid search fallback (vector-only for v1)
- Soft-delete vacuum job (manual SQL documented in README; automation deferred)
- Metrics (Prometheus) and distributed tracing (OpenTelemetry) — Phase 9

## 3. Locked decisions (from brainstorm)

Nine decisions made via clarifying questions before design proposal. These are normative for this phase:

| # | Decision | Rationale |
|---|---|---|
| 1 | **Embedding:** Ollama sidecar default + pluggable interface | Matches CLI default (vector parity), zero external deps for self-host, opt-in to API providers via env |
| 2 | **Workspace isolation:** per-row `workspace_id` + tenant-scoped DB helper | Simplest reasoning surface; reversible to RLS in Phase 9 if compliance demands |
| 3 | **Ingest model:** synchronous (POST returns when searchable) | Single-team self-host scale; no queue/worker complexity |
| 4 | **LLM:** pluggable interface, Anthropic adapter ships first | Anthropic prompt caching cuts cost ~90% on the static system prompt |
| 5 | **Tests:** testcontainers for real Postgres+pgvector | Real DB catches schema and pgvector issues mocks miss |
| 6 | **Libraries:** zod + pino + vitest + @fastify/swagger | Conventional Fastify+Drizzle stack; zod is Fastify's first-class validator |
| 7 | **Key hashing:** argon2id via @node-rs/argon2 | Modern KDF; native Rust binding (no compile step) |
| 8 | **Search ranking:** in-SQL via pgvector + computed columns | One query, faster, scales linearly with index |
| 9 | **`/v1/ask` response:** batch for v1, SSE deferred | Streaming is a UX-only win for browser clients (Phase 9 dashboard) |

**Boundary rules:**
- `embedding/` and `llm/` are pluggable via interfaces — one provider per file + factory file picks at runtime
- `db/client.ts` exposes `withWorkspace(workspaceId, fn)` — every multi-tenant query goes through it
- `format/` parsing is the only place that touches `spec/schemas/`
- Each route file exposes a single Fastify plugin; `app.ts` registers them all
- `src/server.ts` is the only file that calls `.listen()` — `app.ts` is reusable for tests via `app.inject()`

## 5. Postgres schema

Drizzle schemas in `server/src/db/schema/`. Every multi-tenant table carries `workspace_id` (denormalized for partial-index performance and tenant-helper simplicity).

```sql
CREATE TABLE workspaces (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug         text NOT NULL UNIQUE,                    -- matches tolvi-format-v1 slug regex; max 80
  name         text NOT NULL,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE repos (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id  uuid NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  slug          text NOT NULL,                          -- repo slug from .vault-meta.json
  remote_url    text,
  created_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (workspace_id, slug)
);
CREATE INDEX repos_workspace ON repos (workspace_id);

CREATE TABLE documents (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id    uuid NOT NULL REFERENCES workspaces(id),
  repo_id         uuid NOT NULL REFERENCES repos(id) ON DELETE CASCADE,
  doc_type        text NOT NULL,                        -- 'decision' | 'session' | 'pattern'
  path            text NOT NULL,                        -- relative path within vault
  slug            text NOT NULL,                        -- date-stripped slug per spec §7
  status          text NOT NULL DEFAULT 'active',       -- one of the six values per spec §6
  title           text NOT NULL,
  body            text NOT NULL,                        -- raw markdown body (frontmatter stripped)
  frontmatter     jsonb NOT NULL DEFAULT '{}'::jsonb,
  date            date,                                 -- nullable: patterns have no date
  content_hash    text NOT NULL,                        -- sha256 of normalized body+frontmatter (idempotency key)
  deleted_at      timestamptz,                          -- soft delete
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now(),
  UNIQUE (repo_id, path)
);
CREATE INDEX documents_workspace_status   ON documents (workspace_id, status) WHERE deleted_at IS NULL;
CREATE INDEX documents_workspace_doc_type ON documents (workspace_id, doc_type) WHERE deleted_at IS NULL;
CREATE INDEX documents_repo               ON documents (repo_id) WHERE deleted_at IS NULL;
CREATE INDEX documents_slug               ON documents (workspace_id, slug);

CREATE TABLE chunks (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id    uuid NOT NULL REFERENCES workspaces(id),
  document_id     uuid NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
  position        int NOT NULL,                         -- order within document (0-based)
  content         text NOT NULL,                        -- the chunk text (what gets embedded)
  embedding       vector(768),                          -- nomic-embed-text default; configurable per-deployment
  heading_path    text[],                               -- breadcrumb of headings (informative for citations)
  UNIQUE (document_id, position)
);
CREATE INDEX chunks_embedding_hnsw ON chunks USING hnsw (embedding vector_cosine_ops);
CREATE INDEX chunks_workspace      ON chunks (workspace_id);

CREATE TABLE api_keys (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id    uuid NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE,
  key_hash        text NOT NULL,                        -- argon2id hash
  key_prefix      text NOT NULL,                        -- first 8 chars of plaintext, for UI display
  name            text NOT NULL,
  scopes          text[] NOT NULL DEFAULT ARRAY['ingest','search']::text[],
  created_at      timestamptz NOT NULL DEFAULT now(),
  last_used_at    timestamptz,
  revoked_at      timestamptz
);
CREATE INDEX api_keys_workspace ON api_keys (workspace_id) WHERE revoked_at IS NULL;
CREATE INDEX api_keys_prefix    ON api_keys (key_prefix);
```

**Schema notes:**
- `vector(768)` is the default; an `EMBEDDING_DIM` env var is the only thing the Drizzle migration reads when generating the column. Changing it requires a migration regen + reindex (documented in README).
- `content_hash` is the idempotency key for `POST /v1/sync`. Re-POSTing the same content is a no-op.
- `key_prefix` enables fast lookup before the expensive argon2 verify. Look up by `(workspace_id, key_prefix)` first → narrow to <5 rows → argon2 verify each.
- Soft-delete on `documents`. Hard-delete cascades to chunks. Manual vacuum job documented in README.

**Migration discipline:** every Drizzle migration includes a working `down()`. Migration files committed alongside their schema change. CI runs `drizzle-kit check` to ensure the schema/migration pair is consistent.

## 6. API surface

All endpoints under `/v1`. JSON in/out. All authenticated except `GET /healthz` and `GET /readyz`. Errors follow `{ error: { code, message, details? } }`.

OpenAPI spec generated from Fastify route schemas at build time → `spec/openapi.json` (committed). CI fails if the committed file drifts from regenerated form.

### Auth model

Header: `Authorization: Bearer tlv_<key>`. Server splits on `_`, looks up by `key_prefix`, verifies the rest with argon2id. On success, sets `request.workspaceId`.

Token format: `tlv_` + 32 random bytes (base64url) → ~46 chars total. Prefix = first 8 chars after `tlv_`.

### Endpoints

**`GET /healthz`** (no auth) — `{ ok: true, version: "0.1.0" }`. Liveness only.

**`GET /readyz`** (no auth) — 200 if Postgres + embedding provider both reachable, else 503. Used by docker-compose / k8s probes.

**`POST /v1/documents`** — ingest a single doc (sync; returns when searchable).

```jsonc
// Request
{
  "repo": "tolvi",
  "path": "decisions/2026-04-12-foo.md",
  "content": "---\ntags: ...\n---\n\n## Why\n..."
}
// Response 200 (created/updated; idempotent on content_hash)
{
  "document": { "id", "repo_id", "doc_type", "slug", "status", "title",
                "content_hash", "chunks": 4, "embedded_at": "2026-05-10T..." }
}
// Response 200 (no-op when content_hash matches existing)
{ "document": { ... }, "unchanged": true }
// Response 400 — validation failure against tolvi-format-v1
{ "error": { "code": "format_validation_failed", "message": "...",
             "details": [{ "instancePath": "/status", "message": "..." }] } }
```

**`GET /v1/documents`** — list with filters.

Query: `?repo=<slug>&doc_type=decision&status=active,in-progress&limit=50&cursor=<opaque>`

```jsonc
// Response 200
{
  "documents": [ { "id", "repo_id", "doc_type", "slug", "status", "title", "date", "updated_at" } ],
  "next_cursor": "..." | null
}
```

**`GET /v1/documents/:id`** — fetch one with full body + frontmatter + chunk metadata.

```jsonc
{
  "document": { "id", "repo_id", "doc_type", "slug", "status", "title", "body", "frontmatter",
                "date", "content_hash", "created_at", "updated_at",
                "chunks": [{ "position", "content", "heading_path" }] }
}
```

**`DELETE /v1/documents/:id`** — soft delete. 204. Subsequent `GET /:id` returns 404.

**`POST /v1/search`** — semantic + filter search.

```jsonc
// Request
{
  "query": "how do we handle migrations",
  "limit": 10,
  "filters": {
    "repo": "tolvi" | ["tolvi", "tolvi-cli"] | null,
    "doc_type": ["decision", "pattern"] | null,
    "status": ["active"] | "any" | null
  }
}
// Response 200
{
  "results": [
    {
      "document_id": "...",
      "doc_type": "decision",
      "slug": "idempotent-migrations",
      "title": "Idempotent migrations",
      "score": 0.842,                        // weighted (cosine × recency × session)
      "raw_similarity": 0.91,                // unweighted cosine
      "matched_chunk": {
        "position": 2,
        "content": "...",
        "heading_path": ["Idempotent migrations", "When to use"]
      }
    }
  ],
  "total": 17                                // total candidates considered
}
```

**`POST /v1/ask`** — search + LLM synthesis with citations.

```jsonc
// Request — same as /v1/search plus optional model override
{ "query": "...", "filters": { ... }, "model": "claude-sonnet-4-7" | null }
// Response 200
{
  "answer": "Migrations should be idempotent... See [[idempotent-migrations]] for the canonical pattern.",
  "citations": [
    { "slug": "idempotent-migrations", "doc_type": "pattern", "document_id": "..." }
  ],
  "search_results": [ ... same shape as /v1/search.results ],
  "model": "claude-sonnet-4-7",
  "tokens": { "input": 1820, "output": 142, "cache_read": 1612 }
}
```

**`POST /v1/sync`** — batch ingest from CLI publish. Up to 500 docs per request; per-doc results allow partial success.

```jsonc
// Request
{
  "repo": "tolvi",
  "documents": [
    { "path": "decisions/...", "content": "---\n..." },
    // ... up to 500
  ]
}
// Response 200
{
  "results": [
    { "path": "decisions/...", "status": "created" | "updated" | "unchanged" | "failed",
      "document_id": "..." | null,
      "error": null | { "code": "...", "message": "..." } }
  ],
  "summary": { "created": 3, "updated": 7, "unchanged": 12, "failed": 0 }
}
```

Sync runs documents serially within a request (Ollama is the bottleneck; parallelism just queues at it).

**`GET /v1/repos`** — list repos in workspace.

```jsonc
{ "repos": [ { "id", "slug", "remote_url", "document_count", "last_synced_at" } ] }
```

### Error codes

| Code | When |
|---|---|
| `unauthorized` | Missing/invalid bearer token |
| `forbidden` | Token valid but lacks required scope |
| `format_validation_failed` | Doc didn't match `tolvi-format-v1` schemas |
| `not_found` | Doc/repo doesn't exist or was deleted |
| `rate_limited` | Reserved (Phase 9); not enforced in v1 |
| `embedding_unavailable` | Embedding provider unreachable; returns 503 |
| `llm_unavailable` | LLM provider unreachable; returns 503 |
| `internal_error` | Unexpected; logged with request_id |

## 7. Ingest pipeline

`POST /v1/documents` and `POST /v1/sync` both run `ingestDocument(workspaceId, repoSlug, path, content)`:

1. **Parse frontmatter.** Read `---\n...\n---` block; rest is body.
2. **Detect doc type** from path prefix (`decisions/` | `sessions/` | `patterns/`). Reject unknown.
3. **Validate frontmatter** against `spec/schemas/{decision,session,pattern}.json` via Ajv. On failure → `400 format_validation_failed` with the Ajv error path.
4. **Validate filename** — slug regex from spec §4 (`^[a-z0-9]([a-z0-9-]*[a-z0-9])?$`), 80-char cap on slug component, ISO-8601 date prefix check where applicable.
5. **Compute `content_hash`** — SHA-256 of canonicalized form (sorted-keys JSON of frontmatter + body trimmed of trailing whitespace). Stable across formatting noise.
6. **Upsert repo** — get-or-create `(workspace_id, repoSlug)`.
7. **Upsert document** — `INSERT ... ON CONFLICT (repo_id, path) DO UPDATE`. If `content_hash` matches existing → return `unchanged: true`, skip embedding.
8. **Chunk body** — walk markdown AST via `remark` (no transforms). Produce ≤512-token chunks split at heading boundaries when possible, with `heading_path` breadcrumbs preserved. Sliding window of 64 tokens at chunk boundaries to preserve context. Token count via lightweight char-based heuristic (real tokenizer overkill).
9. **Embed chunks** — single batched call to `EmbeddingProvider.embed(texts: string[])`. On provider unreachable → `503 embedding_unavailable`, document is rolled back (no partial state).
10. **Replace chunks** — `DELETE FROM chunks WHERE document_id = $id; INSERT ...` wrapped with the document update in a single transaction.
11. **Return** the document row with `chunks: <count>`.

**Atomicity:** document update + chunk replace happen in one transaction. If embedding fails, nothing changed. Typical decision (300 words, 2-3 chunks) → ~150ms Postgres + ~800ms Ollama embedding = ~1s round-trip. Tolerable for sync POST.

## 8. Search pipeline

`POST /v1/search` → `search(workspaceId, query, filters, limit)`. Single SQL query after embedding the query (one extra Ollama call):

```sql
WITH query_embedding AS (SELECT $1::vector AS v)
SELECT
  d.id              AS document_id,
  d.doc_type,
  d.slug,
  d.title,
  c.position        AS chunk_position,
  c.content         AS chunk_content,
  c.heading_path,
  1 - (c.embedding <=> q.v)                                                    AS raw_similarity,
  -- recency: floor 0.8 + 0.2 decay over 180-day half-life, per spec §9
  (1 - (c.embedding <=> q.v)) *
    (0.8 + 0.2 * exp(-EXTRACT(EPOCH FROM (now() - COALESCE(d.date::timestamp, d.created_at))) / 86400.0 / 180.0)) *
    -- session down-weight 0.7, per spec §9
    CASE WHEN d.doc_type = 'session' THEN 0.7 ELSE 1.0 END                     AS score
FROM chunks c
JOIN documents d ON d.id = c.document_id
JOIN repos r     ON r.id = d.repo_id, query_embedding q
WHERE d.workspace_id = $2
  AND d.deleted_at IS NULL
  AND d.status = ANY($3::text[])
  AND ($4::text[] IS NULL OR r.slug = ANY($4))
  AND ($5::text[] IS NULL OR d.doc_type = ANY($5))
ORDER BY score DESC
LIMIT $6;
```

**Notes:**
- One row per chunk match. Application-side dedupe to one result per document (highest-scoring chunk wins). Cheaper than `DISTINCT ON` for v1 volumes.
- Status filter is a parameter array; route layer fills in spec defaults `['active', 'in-progress', 'historical']` when caller doesn't override.
- HNSW index on `chunks.embedding` makes the cosine scan log-time. Other filters are post-cosine.
- Score formula constants live in `search/ranking.ts` as named constants. Single edit point if spec evolves to v1.x.

## 9. Ask pipeline

`POST /v1/ask` → `ask(workspaceId, query, filters, model)`:

1. **Search** — call `search(...)` with `limit: 8` (8 chunks fit in context comfortably alongside system prompt).
2. **Build prompt:**
   - **System prompt** (cached): vault philosophy, citation requirements (`[[slug]]` syntax mandatory per spec §7), output format expectations. Static, doesn't vary per request.
   - **User message:** the query + 8 search chunks, each annotated with `[[slug]]` so the model can cite by slug.
3. **Call LLM** through `LlmProvider.synthesize({ system, user, model })`. Anthropic implementation passes `cache_control: { type: 'ephemeral' }` on the system block. First request caches; subsequent requests within 5 min hit cache (~90% input-cost reduction).
4. **Extract citations** — `ask/citations.ts` regex-scans the answer for `[[slug]]` references, looks each up in the search results to confirm it actually appeared, returns the metadata.
5. **Return** `{ answer, citations, search_results, model, tokens }`.

**Failure modes:**
- LLM unreachable → `503 llm_unavailable`. No partial answer.
- LLM returns answer with hallucinated `[[slug]]` (slug not in search results) → answer returned as-is, but `citations` array only includes verified slugs, AND the answer string has unverified `[[wrong-slug]]` markers replaced with `[unverified citation: wrong-slug]` so consumers don't render broken links.

**Prompt caching scope:** caches are per-API-key + per-organization, so multiple workspaces sharing one Anthropic key DO share the cache. Fine for OSS — self-hosters typically have one key.

## 10. Docker compose

Two-tier:

- **`server/docker-compose.yml`** — for server-only dev work. Postgres + Ollama only.
- **Root `docker-compose.yml`** — the user-facing self-host artifact. Same services + the server image. `docker compose up` and you're done.

## 11. OpenAPI generation

`@fastify/swagger` + `@fastify/swagger-ui` register at app build time. Each route file declares zod schemas for request body + responses; `fastify-type-provider-zod` converts them to the Fastify schema format swagger consumes.

Build script `npm run openapi:dump` boots the app in describe-only mode (no port, no migrations), serializes the swagger document, writes to `spec/openapi.json`. Committed. CI re-runs the dump and fails if the committed file drifts — guarantees the spec is always current.

Swagger UI mounts at `/docs` in dev mode only (env-gated). `npm run dev` gives interactive API docs at `localhost:3000/docs`.

## 12. Test strategy

Three tiers:

- **Unit tests** (`test/unit/`): chunking, content-hash, ranking-formula constants, citation parsing, frontmatter validation. Pure functions, no DB, vitest. ~50ms per file.
- **Integration tests** (`test/integration/`): hit `app` directly via `app.inject()` (Fastify's in-process HTTP), real DB via testcontainers. One container shared across all integration tests in a single suite. Each test runs in a Postgres transaction that rolls back at the end → no inter-test pollution, no truncate dance. ~10s startup, then per-test is fast.
- **E2E tests** (`test/e2e/`): docker-compose-style — full stack including a real Ollama container with `nomic-embed-text` baked in. Slow (~60s per test). Only the happy paths. Run on CI nightly + `npm run test:e2e` locally on demand. Not in the PR-required `npm test`.

`vitest.config.ts` defines `unit` + `integration` projects so each can be run independently.

CI runs unit + integration on every PR. E2E runs on push-to-main.

## 13. Observability + config

- **Config:** `src/config.ts` parses `process.env` through a single zod schema. Required: `DATABASE_URL`, `EMBEDDING_PROVIDER`, `LLM_PROVIDER` (+ provider-specific keys). Defaults: `PORT=3000`, `LOG_LEVEL=info`, `EMBEDDING_DIM=768`, `EMBEDDING_BASE_URL=http://localhost:11434`. Server fails to start with a clear error if any required key is missing — no fallback-to-undefined.
- **Logging:** pino structured JSON; Fastify built-in request logging on. `request_id` propagated through the request scope. Sensitive fields (`Authorization` header, `ANTHROPIC_API_KEY`) added to pino's `redact` list. Readable in dev (`pino-pretty`), JSON in prod.
- **Metrics:** out of scope for Phase 2. Prometheus endpoint deferred to Phase 9.
- **Tracing:** OpenTelemetry deferred to Phase 9. Request IDs in logs cover the v1 debugging story.
